# Phase 1: 数据层与基线验证 — Plan Review

## 1. Summary
Phase 1 的两个计划设计得非常扎实且目标明确。计划 **01-01** 通过建立 `01-01-AUDIT.md` 审计文档，系统性地对比了 Swift 原版与 RN 实现的差异，确保了数据结构的「契约一致性」；计划 **01-02** 则将开发者关心的存储细节文档化，并为 `DATA-02`（数据持久性）提供了具体的人工验证路径。整体方案在不引入复杂测试框架的前提下，通过「静态审计 + 手工冒烟」有效对齐了核心数据模型。

## 2. Strengths
- **审计文档驱动 (Audit-Driven)**：通过专门的 `01-01-AUDIT.md` 记录字段对照表，不仅方便当前验证，也为后续维护者提供了极佳的跨平台实现映射表。
- **语义严谨性**：明确了 Unix 秒的时间戳标准、`1|2` 的类型枚举以及「半开区间」查询逻辑。这些是记账类应用中极易出现 `off-by-one` 误差的关键点。
- **验证闭环**：计划 01-02 强制要求在 `PROJECT.md` 中写入「杀进程再启动」的冒烟步骤，这直接回应了 Core Value 中关于数据稳定性的承诺。
- **质量门禁**：每个计划都包含 `npm run typecheck` 和 `npm run lint` 的回归，确保文档更新过程中没有引入类型违规。

## 3. Concerns
- **`amount` 类型一致性 (MEDIUM)**：
  - 在 `database.ts` 中，`amount` 被定义为 `TEXT`。
  - 在 `models.ts` 中，`Bill` 接口的 `amount` 是 `string | null`。
  - **风险**：如果 Swift 原版使用的是 `Double` 或其他数值类型，且在此后的图表聚合（Phase 3）中涉及大量数值计算，字符串到数字的转换精度需在 `01-01-AUDIT` 中明确（虽然 `money.ts` 可能处理了，但存储层的语义应在此对齐）。
- **`NULL` 语义对照 (LOW)**：
  - `remark` 和 `icon` 在 `Bill` 接口中标记为 `string | null`。
  - **建议**：在 `01-01-AUDIT` 中应确认 Swift 原版中这些字段是否也允许 Optional。如果原版不允许为空而 RN 存了空，可能会导致渲染层的意外崩溃。

## 4. Suggestions
- **扩充审计粒度**：建议在 `01-01-AUDIT.md` 中增加一列 `nullability`，对比 `Optional` 在 Swift 与 TS 中的一致性，防止因 `undefined/null` 导致的运行时错误。
- **明确文件路径描述**：在 01-02 的冒烟步骤描述中，建议在 `PROJECT.md` 中补充说明 `main.db` 在不同平台（iOS/Android 模拟器）的典型沙箱路径前缀，方便开发者使用第三方工具（如 SQLite Browser）进行底层数据核查。
- **语义校验细节**：在验证 `queryBillsInRange` 时，建议额外关注一下本地时区与 UTC 的转换，确保 Unix 时间戳的构造在 `billRepo.ts` 中是基于本地零点还是 UTC 零点。
