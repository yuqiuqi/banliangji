### 3.10bis G4 — GlassEffectUnion（静态合并）

**Apple 公开 API：** [`glassEffectUnion(id:namespace:)`](https://developer.apple.com/documentation/swiftui/view/glasseffectunion(id:namespace:)) 🍎 OFF

**与 §3.10 Container / §3.11 ID 的区别：**

| API | 作用 | 何时用 |
|-----|------|--------|
| `GlassEffectContainer`（§3.10） | 提供**共享采样区**（让多玻璃元素不物理冲突） | 同屏多玻璃，作前置容器 |
| `glassEffectID(_:in:)`（§3.11） | **动态 Morphing** —— 状态切换时形变 | 按钮展开 → 菜单 |
| `glassEffectUnion(id:namespace:)`（本节） | **静态合并** —— 把多个相同 shape/variant 的玻璃融合为**单一形体** | 地图 App「定位 + 图层」两个按钮合并成一块玻璃 |

**必要条件（全部成立才能合并）：**
1. 所有元素位于同一 `GlassEffectContainer`
2. 每个元素 `glassEffectUnion(id: "<同一 id>", namespace: <同一 @Namespace>)`
3. 所有元素的 `Glass` variant 与 shape 相同

**SwiftUI 示例：**
\`\`\`swift
@Namespace var ns
GlassEffectContainer {
  VStack {
    Button(action: {}) { Image(systemName: "square.2.layers.3d") }
      .buttonStyle(.glassProminent)
      .glassEffectUnion(id: "mapTools", namespace: ns)
    Button(action: {}) { Image(systemName: "location") }
      .buttonStyle(.glassProminent)
      .glassEffectUnion(id: "mapTools", namespace: ns)
  }
}
\`\`\`

**RN 工程近似 🛠️ RN-T2：**

```tsx
// src/components/ios/GlassEffectUnion.tsx（Phase 17 新建）
// 策略：在 GlassEffectContainer 内，把要合并的元素放入同一个 View，
//       共享 borderRadius、去除内部 gap、顶部仅保留一条高光 hairline。
function GlassEffectUnion({ children, style }: Props) {
  return (
    <View style={[styles.union, style]}>
      {/* 共享圆角 + 共享采样层（由外层 GlassEffectContainer 提供） */}
      {children}
    </View>
  );
}
// union 样式：overflow: 'hidden', borderRadius: radii.card, 子元素 gap: 0
```

**应用场景：**
- 主 FAB + 展开菜单项（合并为单块胶体）
- Map 界面「定位 + 图层」双键合并
- 工具栏相邻的同类动作按钮组

**禁区：**
- ❌ 跨 `GlassEffectContainer` 使用同一 union id（Apple 明确要求同容器内）
- ❌ 不同 variant 的玻璃（`.regular` 与 `.clear`）合并
- ❌ 合并动态变化的按钮（用 §3.11 Morphing，不用 Union）
```

```markdown
### 3.15 Expo Blur 实战约束（RN 工程必读）

> **为什么单列一节：** 这些不是 Apple 的规则，而是 `expo-blur` 在 Expo 54 / RN 0.81 下的**已知工程限制**。违反这些约束，v1.2 §3.6–§3.14 的所有近似方案都会「看似正确但实际失效」。

**约束 1 — BlurView 对动态内容的 render 顺序问题** 🛠️ RN-T1 📦 expo-blur

官方文档原话（[Expo BlurView docs](https://docs.expo.dev/versions/latest/sdk/blur-view/)）：

> "If the `BlurView` is rendered **before** dynamic content (e.g., a `FlatList`), the blur may not update as the underlying content changes."

**落地规则：**
- 在 `View` 组件树中，**`BlurView` 必须渲染在动态内容之后**（即 z-index 更高，绝对定位在 FlatList 上方）
- 推荐结构：
  ```tsx
  <SafeAreaView>
    <FlatList data={...} />                        {/* 先 */}
    <BlurView intensity={60} style={styles.tab}/>  {/* 后，绝对定位在顶部/底部 */}
  </SafeAreaView>
  ```
- 禁区：不要把 `BlurView` 作为 `FlatList` 的 `ListHeaderComponent`（blur 不会随滚动内容更新）

**约束 2 — BlurView 并发动画性能** 🛠️ RN-T1

- `intensity` 通过 `Reanimated` 动画驱动时，**单屏并发 ≤ 2 个**（超过会掉帧）
- 复杂场景用 `opacity` 或 `scale` 动画替代 `intensity` 动画

**约束 3 — Android 回退必须** 🛠️ RN-T1

- `BlurView` 在 Android 性能差 / 视觉不等价
- 必须用 `Platform.OS === 'ios' ? <BlurView /> : <SolidFallback />` 模式
- `SolidFallback` 使用 `colors.surface` + `hairlineBorder`（见 §3.3 RN 工程近似）

**约束 4 — BlurView 嵌套限制** 🛠️ RN-T1

- 嵌套 `BlurView`（玻璃内玻璃）在 iOS 会出现灰雾 —— 这与 SwiftUI「玻璃不能采样玻璃」是同一物理约束的 RN 版本
- 必须用 §3.10 `GlassEffectContainer` 模式（单外层 Blur，内层子元素仅 `tint` / `borderRadius`）

**约束 5 — iOS 低端机 GPU 预算**

- iPhone 12 mini / SE 等非 ProMotion 机型：同屏 `BlurView` ≤ 3 个
- ProMotion（iPhone 13 Pro+）可放宽到 5 个，但不建议
- 超标的场景回退为 §3.3 `.identity` 实色

**约束 6 — reducedTransparency 系统开关**

- iOS 系统层面开启「降低透明度」时，所有 `BlurView` 自动回退为 opaque
- `expo-blur` 不需要 app 层代码处理（自动适配）
- 但本仓库 `useReduceTransparency` hook（Phase 11 实现）可用于**额外场景**（如文字阴影加深）
```
</action>
